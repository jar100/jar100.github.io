오늘의 공부 20181107

  1. 객체지향의 사실과 오해 정리

객체지향 애플리케이션은 실세계를 모방 하는게 아니다.
실세계의 방화벽은 불을 막는 것이지만 컴퓨터세계에서는 외부의 침입을 막는 보안 프로그렘인 것 처럼 실세계를 모방 하는 것이 아닌 실세계를 참고 해 새로운 객체를 의인화 해 창조해 내는 것 이다.

캡슐화(encapsulation) 자율성(autonomous) 협력(collaboration)

  1. 메세지를 줘 자기자신이 동작하게 만들어라
  2. 내부의 내용은 외부에서 알지 못하게 해라.
  3. 객체간 서로 협력하게 해라.

우리가 커피를 주문할 때 커피를 만드는 법을 몰라도 시킬 수 있듯이 메세지만 보내면 커피를 받을 수 있게 구현하면 된다.

객체

상태(state) 행동(behavior) 를 지닌 실체
다른 객체가 무엇(what)을 수행하는지는 알 수 있지만 어떻게(how) 수행하는지에 대해서는 알 수 없다.

  • 객체의 상태는 단순한 값과 객체의 조합으로 표현할 수 있다. 모든 상태 특징을 통틀어 객체의 프로퍼티(property) 라고 한다. 프로퍼티는 정적이다. 하지만 프로퍼티 값(property value)는 동적이다.

상태

상태는 객체의 현제 상태를 정의 하는것
인스턴스 변수 라 생각하면 될듯, 프로퍼티
ex) int value = 10;
value -> 프로퍼티
10 -> 프로퍼티 벨류

행동

행동은 객체의 상태에 영향을 받고 , 객체의 상태를 변경 시킨다.

식별자

식별자는 어떤 객체를 다른 객체와 구분하는 데 사용하는 객체의 프로퍼티다. 개발자가 임의로 만드는 패키지, 클래스, 메소드, 변수 이름을 의미
ex) user 라는 객체를 구분 할 때는 id 로 구분하면 id 가 식별자

역할

역할은 대체 가능성을 의미한다.
한 객체가 여라가지의 역할을 가질 수 있다.
ex) 나는 대한민국의 시민의 역할을 갖을 수 있고 아버지의 아들이라는 역할을 갖을 수 있다.

책임

객체에게 책임을 위임(delegation)하라.
객체가 바뀌더라도 그 객체가 담당하고 있는 책임만 다할 수 있다면 된다.
책임을 부여할 때는 추상적이면서도 명확하게 제시

  • 너무 추상적이면 안되고 협력을 통해 책임을 받은 객체가 자율적으로 반응 할 정도의 추상적
  • 어떻게(how)가 아니라 무엇(what)을 하는지만 결정
    다형성 : 동일한 요청에 대해 서로 다른 방식으로 응답할 수 있는 능력
    ex) 도형에게 그려라 라고 요청을 하면 삼각형은 삼각형을 그리고 사각형은 사각형을 그려 응답한다.

협력

객체간 협력을 통하여 애플리케이션을 구현한다. 협력을 생각하고 코드를 구현하는게 좋다.
객체지향의 핵심은 클래스가 아니라 적절한 역할을 수행하는 역할 간의 유연하고 견고한 협력 관계를 구축하는 것이다.
객체의 역할, 책임, 협력에 집중하자.

갞체지향 설계 기법

  1. 책임주도 설계

    • 시스템이 사용자에게 제공해야 하는 기능인 시스템 책임을 파악한다.
    • 시스템 책임을 더 작은 책임으로 분할한다.
    • 분할된 책임을 수행할 수 있는 적절한 객체 또는 역할을 찾아 책임을 할당한다.
    • 객체가 책임을 수행하는 중에 다른 객체의 도움이 필요한 경우 이를 책임질 적절한 객체 또는 역할을 찾는다.
    • 해당 객체 또는 역할에게 책임을 할당함으로써 두 객체가 협력하게 한다.

  2. 디자인 패턴

  3. 테스트 주도 개발
