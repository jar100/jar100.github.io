step4 하면 서 영속성에 대해 좀더 알게 된거같다.

영구적인 속성이라고 풀이하면 될듯하다.

영속성전이는 영속성이 전이된다. 즉 부모 엔티티가 가지고있는 자식도 부모가 저장되면 같이 저장된다.

spring jpa

Spring Data JPA에서는 쿼리 메소드에 페이징과 정렬 기능을 제공하는 2가지 클래스를 제공한다.

org.springframework.data.domain.Sort : 정렬 기능
org.springframework.data.domain.Pageable : 페이징 기능

Page 클래스가 제공하는 인터페이스
int getNumber(); //현재 페이지
int getSize(); //페이지 크기
int getTotalPages(); //전체 페이지 수
int getNumberOfElements(); //현재 페이지에 나올 데이터 수
long getTotalElements(); //전체 데이터 수
boolean hasPreviousPage(); //이전 페이지 여부
boolean isFirstPage(); //현재 페이지가 첫 페이지 인지 여부
boolean hasNextPage(); //다음 페이지 여부
boolean isLastPage(); //현재 페이지가 마지막 페이지 인지 여부
Pageable nextPageable(); //다음 페이지 객체, 다음 페이지가 없으면 null
Pageable previousPageable(); //다음 페이지 객체, 이전 페이지가 없으면 null
List getContent(); //조회된 데이터
boolean hasContent(); //조회된 데이터 존재 여부
Sort getSort(); //정렬정보

상태기반으로 할 때 사용하는게 쿠키,세션

무상태는 한번 요청을 보내고 받으면 연결이 끊어진다.
html은 기본적으로 무상태 기반이라 쿠키 라는 것을 이용 응답이 올때 지속적으로 보내 상태를 유지 시키는 방법으로 상태를 구현
하지만 무상태에서 상태를 구현한것이라 업데이트 된것이 바로바로 클라이언트 눈에 보이지 않는다.(새로고침을 해야한다.)
ex) 네이버 야구 10초마다 서버에 요청해 정보를 받아온다.

상태는 나한테 연결되있는 클라이언트의 정보를 가지고있다가 서버에서 변경사항을 바로 클라이언트에게 알려준다.
즉 상태기반 클라이언트의 정보를 계속 가지고있으면서 반영해준다.
http가 발전하면서 WebSocket 이 나오면서 상태기반으로 구현할 수 있게 됬다.
하지만 상태 기반으로 웹페이지 전부를 구현하게 되면 한번에 많은 유저데이터를 지속해서 가지고 있어야한다. (서버에 부담이 커진다.)
그래서 요즘 웹페이지는 특정 부분은 WebSocket 을 이용해 상태로 구현 하고 나머지 부담이 많은 부분은 무상태로 구현하는 것을 지향한다.
ex) 페이스북 dm , 웹페이지의 쪽지나 알람기능

무상태 : polling

cascade/orphanRemoval

cascade : 폭포수
cascade 를 지정해주면 부모 데이터 베이스가 삭제되면 자식도 같이 삭제되게 할 수 있다.

orphanRemoval=true

  • Emp 엔티티가 삭제될 때 참조가 끊어진 연관된 Addr 엔티티도 삭제하라는 의미이며 DB에서도 삭제되는데 참조(연결)가 끊어진 Addr 객체는 DB에서도 삭제된다는 뜻이다.

cascade=CascadeType.REMOVE

  • Emp 엔티티가 삭제될 때 연관된 Addr 엔티티도 삭제하라는 의미이며 DB에서도 삭제된다.

연결이 끊어진다고 해서 자동 삭제되는 것은 아니고 명시적으로 연관 엔티티가 삭제될 때 같이 삭제하라는 영속성 전이와 관련된 옵션이다.

스프링 애플리케이션 사용시 빌드를 자동으로 해주는 툴

  1. 그래들에 devtools 컴파일
    • compile(‘org.springframework.boot:spring-boot-devtools’)
  2. 인텔리제이 setting 의 compile 에 build project automatically 를 채크
  3. Registry 에 commpiler.automake.allow.when.app.running 채크
  4. application.properties 작성 (안해도 디폴트 true 로 동작하는것 같다. )
    • spring.devtools.livereload.enabled=true
    • spring.devtools.restart.enabled=true
    • spring.thymeleaf.cache=false

url 기능

GET /todos = List all todos
POST /todos = Create a new todo
GET /todos/:id = Get a todo
PUT /todos/:id = Update a todo
DELETE /todos/:id = Delete a todo and its items
GET /todos/:id/items = Get a todo item
PUT /todos/:id/items = Update a todo item
DELETE /todos/:id/items = Delete a todo item
