오늘의 공부 20181109

데이터베이스

RDBMS

안전(Safe) 하다.

  • 데이터를 영구적으로 저장
  • 은행, 증권회사에서 사용
  • 트랜잭션의 보장 ex) a에서 b로 돈을 옮기면 a는 줄어들고 b는 많아진다.

안정적(Reliable) 이다.

  • 고가용성(HA): 1년에 99.999% 시간 동작을 보장.

효율적(EFFicient) 이다.

  • 다사용자/ 대용량 데이터를 동시에 처리 가능
  • 한 명이 한 번씩 실행한 것과 비슷한 결과를 보장

편리(conveinent) 하다.

  • 선언적 언어인 SQL을 사용, 알고리즘에 대해 신경쓸 필요 없음 what 만 신경쓰고 how 를 신경 쓸 필요가 없다.
  • 데이터 독립성: 물리적인 저장구조에 대해 신경쓸 필요가 없음

noSQL (not only SQL)

RDBMS 가 아닌 데이터베이스
더 빠르다.
더 사용이 쉽다.
매우 큰 데이터(PB)를 저장할 수 있다.

오늘의 공부 20181107

  1. 객체지향의 사실과 오해 정리

객체지향 애플리케이션은 실세계를 모방 하는게 아니다.
실세계의 방화벽은 불을 막는 것이지만 컴퓨터세계에서는 외부의 침입을 막는 보안 프로그렘인 것 처럼 실세계를 모방 하는 것이 아닌 실세계를 참고 해 새로운 객체를 의인화 해 창조해 내는 것 이다.

캡슐화(encapsulation) 자율성(autonomous) 협력(collaboration)

  1. 메세지를 줘 자기자신이 동작하게 만들어라
  2. 내부의 내용은 외부에서 알지 못하게 해라.
  3. 객체간 서로 협력하게 해라.

우리가 커피를 주문할 때 커피를 만드는 법을 몰라도 시킬 수 있듯이 메세지만 보내면 커피를 받을 수 있게 구현하면 된다.

객체

상태(state) 행동(behavior) 를 지닌 실체
다른 객체가 무엇(what)을 수행하는지는 알 수 있지만 어떻게(how) 수행하는지에 대해서는 알 수 없다.

  • 객체의 상태는 단순한 값과 객체의 조합으로 표현할 수 있다. 모든 상태 특징을 통틀어 객체의 프로퍼티(property) 라고 한다. 프로퍼티는 정적이다. 하지만 프로퍼티 값(property value)는 동적이다.

상태

상태는 객체의 현제 상태를 정의 하는것
인스턴스 변수 라 생각하면 될듯, 프로퍼티
ex) int value = 10;
value -> 프로퍼티
10 -> 프로퍼티 벨류

행동

행동은 객체의 상태에 영향을 받고 , 객체의 상태를 변경 시킨다.

식별자

식별자는 어떤 객체를 다른 객체와 구분하는 데 사용하는 객체의 프로퍼티다. 개발자가 임의로 만드는 패키지, 클래스, 메소드, 변수 이름을 의미
ex) user 라는 객체를 구분 할 때는 id 로 구분하면 id 가 식별자

역할

역할은 대체 가능성을 의미한다.
한 객체가 여라가지의 역할을 가질 수 있다.
ex) 나는 대한민국의 시민의 역할을 갖을 수 있고 아버지의 아들이라는 역할을 갖을 수 있다.

책임

객체에게 책임을 위임(delegation)하라.
객체가 바뀌더라도 그 객체가 담당하고 있는 책임만 다할 수 있다면 된다.
책임을 부여할 때는 추상적이면서도 명확하게 제시

  • 너무 추상적이면 안되고 협력을 통해 책임을 받은 객체가 자율적으로 반응 할 정도의 추상적
  • 어떻게(how)가 아니라 무엇(what)을 하는지만 결정
    다형성 : 동일한 요청에 대해 서로 다른 방식으로 응답할 수 있는 능력
    ex) 도형에게 그려라 라고 요청을 하면 삼각형은 삼각형을 그리고 사각형은 사각형을 그려 응답한다.

협력

객체간 협력을 통하여 애플리케이션을 구현한다. 협력을 생각하고 코드를 구현하는게 좋다.
객체지향의 핵심은 클래스가 아니라 적절한 역할을 수행하는 역할 간의 유연하고 견고한 협력 관계를 구축하는 것이다.
객체의 역할, 책임, 협력에 집중하자.

갞체지향 설계 기법

  1. 책임주도 설계

    • 시스템이 사용자에게 제공해야 하는 기능인 시스템 책임을 파악한다.
    • 시스템 책임을 더 작은 책임으로 분할한다.
    • 분할된 책임을 수행할 수 있는 적절한 객체 또는 역할을 찾아 책임을 할당한다.
    • 객체가 책임을 수행하는 중에 다른 객체의 도움이 필요한 경우 이를 책임질 적절한 객체 또는 역할을 찾는다.
    • 해당 객체 또는 역할에게 책임을 할당함으로써 두 객체가 협력하게 한다.

  2. 디자인 패턴

  3. 테스트 주도 개발

느낀점

다시한번 읽고 읽은 것을 바탕으로 정리를 해봤다. 아직 파트 6,7 에 관한 유스케이스 책임주도설계 방법 디자인페턴 등등 은 이해가 조금 안간다.
책을 읽으면서 조금 더 발전하는게 느껴진다. 다음에 또 읽어보면 6,7 장도 이해가 됬으면 좋겠다.
정말 나중에 한번 더 읽어 보고싶은 책인거 같다.

throw throws 차이

throw는 현재 블록에 exception 발생

compiletime, runtime Exception 차이

둘다 예외 처리하는것 compiletime throws 를 꼭 써야한다. runtime 안써도 됨

  • 읽는사람을 위해 throws 를 쓰면 좋다. throws 를 적으면 어떤 애러들이 발생하는지 바로 알 수 있다.
  • runtime 자주 사용하고 반드시 try catch를 강제하고싶을때만 compiletime 사용하자.
    • 이유는 compiletime을 남발하면 코드가 지저분 해진다.

체크 리스트 *

  1. 상속을 이해했는가?

    • 상속이란 먼저 완성된 클래스 내용을 전부 사용하면서 추가로 새로운 특징이 만들어 졌을 때 중복코드를 제거 하기 위해 사용

  2. 추상 클래스와 추상 메서드에 대해 이해했는가?

    • 추상 클래스와 메서드는 미리 추상적으로 선언만 해둔 것 추상 클래스를 사용하면 각 매서드의 공통점은 상속시키고 차이점은 추상메서드로 강제 해 자식클래스들이 꼭 정의하게 할 수 있다.

  3. this와 super 키워드에 대해 알고 있는가?

    • this 는 본인, 자기 자신, class 내부의 인스턴스를 가리킬 때 사용
    • super 는 부모의 인스턴스를 나타낼 때 사용 굳이 붙이지 않아도 알아서 잡아줌 하지만 같은 명칭이 사용되면 super와 this 로 구분

  4. 인터페이스를 이해했는가?

    • 외부에 값을 전달하기 위해 미리 정해 놓은 것
    • 미리 규격을 약속해 놓으면 협업이나 특정부분 모듈을 바꿀 때 유연하게 사용 할 수 가 있다.

  5. 업캐스팅에 대해 이해했는가?

    • 상위 클래스로 형 변환 하는것
    • 아직 잘 이해를 못했다. 더 공부하면 좋겠다. 다음에 업캐스팅에 더 공부 해보자.

  6. 다운캐스팅에 대해 이해했는가?

    • 하위 클래스로 형변환 하는것
    • 다음에 나오면 같이 더 공부 해봐야 겠다.

  7. ArrayList와 LinkedList의 차이점에 대해 이해했는가?

    • arrayList는 순서대로 저장하는 느낌
    • LinkedList 리스트 에게 다음 순서 리스트 정보를 적어 순서를 이동 시킴
    • 만들고 찾는건 arrayList 가 빠르지만 특정값을 수정하거나 순서를 추가할 땐 LinkedList가 더 좋다.

  8. equals, hashCode, toString 세 메소드의 용도에 대해 알고 있는가?

    • object class 메소드 셋다 class 를 생성하고 비교 출력 할 때 override 해서 자주 사용
    • equals 는 값을 비교하는 것
    • hashCode 는 기본은 주소값을 리턴한다.
    • toString 은 객체가 출력할 기본 출력값을 String으로 담고있다.

  9. List와 Set의 차이점에 대해 알고 있는가?

    • list 는 중복값 허용 값은 get로 뽑아냄
    • set 은 중복값 불가 값을 뽑아 올 때 iterator를 사용 해야함.

  10. 인스턴스를 비교할 때 ==와 equals()를 사용할 때의 차이점에 대해 알고 있는가?

    • 기본적으로는 equals는 == 로 비교하는데 equals는 override 해서 다른값을 비교 하게 할 수 있다.
    • String 같은 객체들은 equals로 비교 해야함.

  11. 상속과 인터페이스의 차이점과 상속과 인터페이스가 필요한 이유에 대해 설명하시오. *

    • 상속은 부모클래스에 선언된 모든것을 가져오는 것
    • 인터페이스는 특정 메소드를 꼭 만들라고 강제 하는 것

  12. 이번 미션에서 공부한 자바 문법 중에서 학습하기 가장 어려웠던 것은 어떤 것인가요? *

    • 인터페이스와 다형성이 어려웠다.

  13. 그 외에 학습하기 어렵거나 시간이 오래 걸린 것은 무엇이고, 어떻게 극복했나요? *

    • 객체 설계 하는게 어려웠다. 객체는 파도 파도 새로운거같다. 오늘은 여기까지 하고 내일 더 공부 해봐야겠다.
